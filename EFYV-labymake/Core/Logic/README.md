# Application Logic

[Core](../README.md) | [Repository](../../README.md) | [Models](../Models/README.md) | [Tools](../Tools/README.md)

This directory contains the host-facing application services. [`DesignerSession`](DesignerSession.cs) is the central facade; the remaining types are independently injectable/testable collaborators.

## Session and history

- `DesignerSession` owns the active project, selected animation/frame, active tool, dirty version, validation result (computed on demand — see *Deferred validation*), preview, autosave, live-debug controller, and bounded command history.
- Pointer down captures the frame and active tool. Drag/up continue against that stable pair, so changing UI selection mid-gesture does not redirect the edit.
- Undo/redo clamp the selection: the current indices are kept while they still address a frame, the frame index is clamped into the selected animation's range, and otherwise the first animation with frames is selected — `CurrentFrame` is null after a history operation only when no animation has frames.
- [`DesignerCommands`](DesignerCommands.cs) stores changed pixel indices or hitbox deltas rather than whole frames. Exceptions and `CancelGesture` restore the captured before-state directly (never by re-diffing), so a tool that structurally mutates `Frame.Layers` mid-gesture rolls back cleanly and only one exception propagates: the tool's own failure, or `FrameEditCommand`'s `InvalidOperationException` when the captured layer no longer occupies its captured slot. Item #6: gestures of the stamp tool additionally capture the frame's attachment list (deep copies); a changed list diffs as full before/after copies (attachments are tiny records) inside the same single gesture command.
- [`CommandManager`](CommandManager.cs) bounds undo/redo by command count and estimated bytes, normalizes invalid estimates, and evicts oldest commands before accounting can overflow. A command that throws during Undo/Redo replay is dropped from both stacks; `HistoryChanged` publishes the truthful counts, then `CommandFailed` reports the dropped command and its fault, and the original exception propagates.
- Animation, frame, layer, property, opacity, visibility, FPS, reorder, generated-animation, and canvas-resize changes are recorded commands.
- Item #10 timing/playback commands: `SetFrameDurationMs` (per-frame override, 0 = inherit FPS), `SetAnimationLoopRange` (both members as ONE command; `loopEnd` may be -1 for "the last frame"), and `SetAnimationPingPong`.
- Item #10 cross-frame layer batches (`AddLayerToAllFrames`, `RemoveLayerFromAllFrames`, `RenameLayerInAllFrames`, `SetLayerVisibilityInAllFrames`) apply one layer operation to EVERY frame of the selected animation as ONE undoable command. Frame keeps owning its own layers (no shared layer identity — the add batch creates one instance per frame); index-addressed variants validate the index against every frame UP FRONT so a batch can never half-apply, and undo restores each frame's original (possibly divergent) name/visibility/instance.

## Selection, floating buffer, and canvas resize

- The session owns transient (non-persisted, non-undoable) selection state: the `SelectionRegion` a selection tool completed, the `FloatingSelection` being previewed, and the copy/paste clipboard. Pointer-down inside the active selection lifts it (cut) into the floating buffer and drags it; `LiftSelection(layer, removeSource)`, `CopySelection`, `PasteClipboard`, `MoveFloating`, `AnchorFloating`, `CancelFloating`, and `ClearSelection` drive the same lifecycle programmatically.
- Only `AnchorFloating` commits — the entire lift-move-anchor interaction becomes ONE sparse `FrameEditCommand` against the capture taken at lift/paste time (pointer-down outside the buffer anchors it; Esc cancels and restores the lifted pixels). Any structural or history mutation cancels an un-anchored buffer (never silently commits it) and clears the region, so a floating capture can never diff against a reshaped frame.
- `ResizeCanvas(width, height, anchor)` is the only supported way to change the canvas of a project with frames: it rebuilds every frame of every animation at the new size with content anchored per the nine-position [`CanvasAnchor`](../Models/CanvasAnchor.cs) (hitboxes translate with the content and clamp to the new canvas in world units) and records the whole rebuild as one undoable command — undo restores the original frame objects. The raw `EFYVProject` canvas setters are internal because they desync existing layer buffers.

## Effects authoring and layer filters (item #7)

- Effect descriptor CRUD: `AddAnimationEffect`, `RemoveAnimationEffect`, and `ReplaceAnimationEffect` are index-addressed undoable commands over `AnimationState.Effects` (cap: `LabyMake.Effect.MaxEffectsPerAnimation`). [`EffectDescriptor`](../Models/EffectDescriptor.cs) instances are immutable, so the commands only insert/remove/swap references; replacing with the same reference is a no-op. Like the palette CRUD, effect operations never touch frames and do NOT cancel an un-anchored floating selection.
- Destructive layer filters: `ApplyBlurFilter(layer, radius)`, `ApplyOutlineFilter(layer, colorRgba)`, `ApplyGlowFilter(layer, colorRgba, radius)`, and `ApplyColorShiftFilter(layer, hueDeltaDegrees, saturationDelta, valueDelta)` run the backend `FastEffects` primitives over ONE layer of the current frame and commit through the same sparse `FrameEditCommand` diff path as pointer gestures — undo restores the exact previous bytes, a no-op filter records nothing (returns `false`). Parameter bounds live in `LabyMake.Filter` (blur radius 1..64, glow radius 0..64, saturation/value deltas in [-1,1], finite hue).
- Region semantics: an active `SelectionRegion` MASKS the write-back (the filter still samples the whole layer — standard region-filter behavior) and the region survives the operation; an un-anchored floating buffer is canceled first (its lifted pixels restored) so the filter sees the true layer bytes. Filters refuse to run mid-gesture and without a current frame.
- [`ToolbarAPI.GetLayerFilters`](ToolbarAPI.cs) is the host-facing filter catalog: `LayerFilterDefinition` entries (kind, display label, which parameters apply, config-owned radius bounds) mapping 1:1 onto the session `Apply*Filter` methods.

## Sub-element attachments (item #6)

- Attachment CRUD: `AddAttachment(name, x, y, zOrder, flipX, flipY)`, `MoveAttachment(index, x, y)`, and `RemoveAttachment(index)` are index-addressed undoable commands over `CurrentFrame.Attachments` (cap: `LabyMake.Attachment.MaxPerFrame`; the [`SubElementAttachment`](../Models/SubElementAttachment.cs) constructor is the validation gate). Like the palette/effect CRUD they never touch pixels and do NOT cancel an un-anchored floating selection, but they DO refuse to run mid-gesture — the stamp tool's attachment mode may be diffing the same list inside the active gesture.
- `ResizeCanvas` shifts attachment anchors with the anchored content (no clamping — an attachment may hang off-canvas; export flattening clips).
- `ProjectValidator` reports `AttachmentLimitExceeded` (per-frame cap) and `InvalidAttachment` (null entry) as structural errors.
- `DesignerSession.Create` accepts an optional [`ISubElementResolver`](ISubElementResolver.cs) (hosts pass their `AssetBankManager`) so export/live-debug can flatten attachments into the published atlas; without one, attachments still export as structured metadata.

## Maps and tilesets (item #5)

- Tileset CRUD: `CreateTileset(tileSize)`, `RemoveTileset`, `AddTilesetTile(name)` (blank), `AddTilesetTileFromCurrentFrame(name)` (captures the flattened top-left TileSize-square of the current frame — the "author a tile like a mini-frame" loop, pairing with the TileMakerTool's wrap-inside-the-cell strokes), `RemoveTilesetTile(index)`, and `RenameTilesetTile(index, name)` — each ONE undoable history entry over the optional [`TilesetSection`](../Models/Tileset.cs). Like the palette CRUD they never touch frames and do not cancel an un-anchored floating selection, but they refuse to run mid-gesture or mid-map-paint.
- Map section lifecycle: `CreateMapSection(mapId, width, height, tilesetName)` / `RemoveMapSection` / `SetMapTilesetName` manage the optional [`MapSection`](../Models/MapSection.cs), all undoable.
- Map editing lands in history and dirties/autosaves the project — the old free-floating `MapTool.TargetMap` path bypassed all three. `SetMapTile(x, y, id)` is one entry per cell (out-of-map returns `false`, surfaced not dropped); `FillMapRect`/`FloodFillMapTiles` run the `FastGridMap` bulk primitives and record ONE sparse diff each; `ApplyMapTool(tool, x, y)` runs a whole MapTool gesture (scatter/tile/noise/automata) with the tool's `TargetMap` pointed at the section grid and its seed synced from `Project.DesignerSeed` (deterministic replay), recording the tile diff PLUS the props the gesture appended as ONE `MapEditCommand` — undo removes exactly the appended prop instances from the swap-list tail and restores the diffed cells.
- Paint strokes batch like pointer gestures: `BeginMapPaint(tileId)` / `PaintMapCell(x, y)` / `EndMapPaint()` commit the whole stroke as ONE sparse entry (first-touch before-values; unchanged strokes record nothing); `CancelMapPaint()` (also `CancelGesture`/Esc) restores every touched cell, and `Undo`/`Redo` auto-cancel an open stroke first. Tile ids are bounded below by `LabyMake.MapDocument.BlankTileId` (-1, the blank cell).
- [`MapRenderer`](MapRenderer.cs) is the host-agnostic map-surface compositor for shell map mode: one TileSize-square cell per map cell — tileset tiles blit their pixels, non-blank ids without tileset pixels fill with a deterministic FNV placeholder color, blank cells stay transparent.
- Export: `ExportEngine.ExportTileset` publishes the tile-sheet `.efyvlaby` (+ tile-ID manifest) and `ExportEngine.ExportMap` the `.efyvmap` binary — see [Export](../Export/README.md).

## Palette and color workflow (item #8)

- The session owns the undoable, index-addressed palette CRUD surface: `AddPalette`, `RemovePalette`, `RenamePalette`, `AddSwatch`, `RemoveSwatch`, `MoveSwatch` — each is one recorded command and marks the project dirty. Palette operations never touch frames, so (unlike frame/layer CRUD) they deliberately do NOT cancel an un-anchored floating selection.
- `TrySelectSwatch(palette, swatch, out rgba)` resolves a swatch, applies it to the active tool when that tool is an `IColorTool`, and records it as recently used; stale indices return `false` without throwing. `NotifyColorUsed(rgba)` pushes into the persisted [`RecentColorRing`](../Models/Palette.cs) and dirty-marks only when the ring actually changed. Recent-color tracking never records history — color selection is not part of the document edit history.
- `ReplaceColor(from, to, allFrames)` is the undoable global color swap: every pixel exactly equal to `from` becomes `to` on EVERY layer (hidden layers included — it is a data operation) of the current frame or of all frames, as ONE history entry built from sparse per-layer index diffs (`ColorSwapCommand`); a layer shared by several in-scope frames is swapped once. Returns the replaced-pixel count; zero matches records nothing. An un-anchored floating selection is canceled (restored) first.
- Palette-constraint mode: with `PaletteConstraintEnabled` and a valid `ActivePaletteIndex`, starting a pointer gesture snaps the active `IColorTool`'s color to the nearest entry of the active palette (metric documented on `Palette.FindNearestIndex`: squared Euclidean over straight RGBA, ties to the lowest index). The snap persistently updates the tool color; eraser/eyedropper/selection tools are unaffected, and a stale index or empty palette silently disables the snap.

## Linked directional authoring (item #33)

- A directional project holds all four facings in ONE project: the ACTIVE facing's animations live in `Project.Animations` (every existing tool/command/undo path keeps operating on the list it already knows) and the other three sets are parked in [`DirectionalState`](../Models/DirectionalState.cs) (`EFYVProject.Directional`; null = plain project). Canvas size, palettes, asset properties, and tileset/map sections are shared across facings.
- `EnableDirectionalAuthoring()` converts a directional-capable project (LivingEntityData family) in place — the existing animations become the facing named by the current facing property (`Entity.DefaultActiveFacing` when absent/invalid) — and `SwitchFacing(facing)` swaps animation sets BY REFERENCE (no pixel copies), keeping the facing property in lockstep. Both are undoable commands; history is strictly LIFO, so a command recorded against one facing's list always replays with that facing swapped back in. In directional mode `SetProperty("facing", ...)` routes through `SwitchFacing`.
- `GenerateMirroredFacing(source, target)` (the Left/Right horizontal pair only) deep-clones every source-facing animation with pixels flipped horizontally, hitboxes X-mirrored in world units, and attachments X-mirrored with `FlipX` toggled, then REPLACES the target facing's set as one undoable command (returns the generated count; both-empty records nothing).
- Whole-project operations that must keep facings consistent — `ResizeCanvas` and `ReplaceColor` over all frames — span every facing via `EFYVProject.EnumerateAllAnimations`.
- `ToolbarAPI.CreateNewLinkedDirectionalProject(assetType)` creates a linked project (active = `DefaultActiveFacing`, other sets empty; null for unknown/non-directional types); `ToolbarAPI.GetFacingOptions(project)` is the host's facing-switcher catalog (non-empty exactly for linked projects).
- ONE export publishes all four facings (see [Export](../Export/README.md)); the parked sets persist in the `.efyvmake` directional section (see [Persistence](../Persistence/README.md)).

## Schema and validation

- [`AssetSchemaService`](AssetSchemaService.cs) turns the shared config manifest into immutable field/type definitions and supports exact-type manifest registration without reflection.
- Item #33b runtime-extensible fields: the `RegisterAssetType(registration, customFields)` overload ADDS custom fields (name + `Float`/`Integer`/`Text` slot kind; caps in `LabyMake.Schema.MaxCustomFieldNameLength`/`MaxCustomFieldsPerType`) to a derived type without any backend config edit. Custom fields become ordinary optional designer fields (label = name, kind-default value, no range/choices) and their values ride the exported `.efyvlaby` properties object as ordinary keys — unknown to the compiled schema manifest by construction, so the Unity importer parks them on `SchemaBackedAssetData`'s string-keyed custom-property store. A custom name may not shadow inherited fields, shared manifest slots, or identity/routing keys; one invalid field rejects the whole registration (nothing lands).
- [`ToolbarAPI`](ToolbarAPI.cs) creates directional/non-directional categories, exposes inspector metadata, normalizes typed values, and enforces ranges, choices, and read-only fields.
- [`ProjectValidator`](ProjectValidator.cs) returns structured issue codes and locations. It checks schema properties, identities/facing, finite values, boss phase thresholds, canvas/atlas limits, animation/frame/layer structure, opacity, hitboxes, effect-descriptor lists (item #7: null entries → `InvalidEffectDescriptor`, per-animation count → `EffectLimitExceeded`; instances are constructor-validated), attachment lists (item #6: null entries → `InvalidAttachment`, per-frame count → `AttachmentLimitExceeded`), and export paths. The atlas budget follows the exporter's actual near-square grid layout (`FastExporter.ComputeAtlasLayout`), not the retired single-row model. The `Structural` scope runs only the canvas/animation/frame/layer/hitbox integrity checks; `DesignerSession.LoadPreview` uses it so schema-level gaps (for example an unnamed entity) never block playback, while export keeps the full scope. Item #33: at EXPORT scope only, a directional project's parked facings are validated too — an empty inactive facing reports `DirectionalFacingIncomplete` (Subject = the facing name) and non-empty parked sets run the same structural checks (each set gets its own atlas budget, matching the per-facing atlases the exporter produces).

## Preview and live debug

- [`PreviewController`](PreviewController.cs) plays immutable composite frames with exact tick accumulation, seek/loop controls, and overflow-safe large elapsed times. Item #10: every frame's duration is its `DurationMs` override or one FPS interval (the accumulator runs in exact tick-times-fps units, so no repeating-fraction drift); looping playback honors the clamped loop range with frames before it as a one-shot intro, `PingPong` bounces direction at the range ends, and huge elapsed times reduce modulo one full loop cycle (state-preserving) instead of stepping frame by frame. Non-looping playback ignores the tags (forward once, pause at the end). `Stop()` on a zero-frame animation keeps the `Empty` state, so `CopyCurrentPixelsTo` keeps throwing its intended `InvalidOperationException`.
- [`LiveDebugController`](LiveDebugController.cs) debounces change notifications, validates after the delay, captures on the supplied synchronization context, and exports snapshots in background work.
- Item #27 live fast path: `NotifyProjectChanged` carries a [`DesignerDirtyScope`](DirtyScope.cs) (default `Everything`) declaring what the edit touched. The controller accumulates the scope across superseded debounce requests since the last successful publish; when the accumulated scope changed no exported pixels or atlas layout (`IsMetadataOnly` — only `Properties`/`Hitboxes` bits), the export takes the metadata-only publish path (no PNG re-encode; see [Export](../Export/README.md)). The scope is captured as late as possible and cleared only for the bits the completed publish covered, so a pixel edit and a hitbox nudge inside one window still republish the PNG. An untagged notification contributes `Everything`, so the fast path never engages by omission.
- [`DebounceScheduler`](DebounceScheduler.cs) abstracts time/delay so autosave and live-debug state machines are deterministic in tests.
- Both controllers publish immutable state snapshots rather than exposing mutable worker state.

## Deferred validation (item #27)

`MarkDirty` no longer runs `ProjectValidator.Validate` synchronously inside the session lock on every command — it sets a dirty flag and passes the command's [`DesignerDirtyScope`](DirtyScope.cs) to the live debug controller. Validation is computed ON DEMAND: `DesignerSessionSnapshot.Validation` is a lazy that resolves through the session (validate-if-dirty, cached behind the flag, outside the gate lock), so publishing a snapshot on every keystroke costs nothing and the (possibly heavy) full validate runs only when a reader actually asks — and never on the synchronous stroke path in a UI host, where snapshots reach the shell on a later message-loop turn. Any reader (`Current`, the shell problems display, the tests) always gets a CURRENT result; `LoadPreview` and export still validate their own scopes eagerly. Per-command dirty scopes are tagged only where a command provably changes no exported pixels (properties, palettes/recent colors, animation playback tags and names, effect descriptors, hitbox-tool gestures); everything else keeps the full-publish default.

## Supporting services

- [`AnimationGeneratorAPI`](AnimationGeneratorAPI.cs) builds walk, eight-octant jitter, bob/breathe, and shake/hit-flash animations while copying base-frame hitboxes. Every generator emits single-layer frames named `Animation.GeneratedLayerName`, and the static `MergeOntoTargetLayer` makes regeneration layer-preserving: only the target layer's pixels are replaced per frame while other layers (manual touch-ups, CLONED for undo isolation), hitboxes, per-frame durations, and playback tags of the existing animation survive; a frame without the target layer gains the generated content at the bottom of its stack, and the generator owns the cycle length (extra existing frames drop). `DesignerSession.GenerateAnimation` routes its replace-in-place path through this merge.
- [`AssetBankManager`](AssetBankManager.cs) atomically stores bounded `.efyvsub` records, skips corrupt entries with a `LoadFailed` event, and extracts flattened canvas regions. Item #6: the writer emits `.efyvsub` version 2 (pivot + default-transform header) while the reader accepts the range `[MinSupportedSubElementFormatVersion .. SubElementFormatVersion]` — a version-1 file loads with the default pivot/transform. It implements [`ISubElementResolver`](ISubElementResolver.cs): `TryResolveSubElement(name)` loads one bank element by file stem, returning `false` for missing names (silently) and for present-but-broken files (after raising `LoadFailed`).
- [`ViewportController`](ViewportController.cs) maps screen/canvas coordinates and renders a flattened frame through backend nearest-neighbor scaling. An overload composites the session's `FloatingSelection` over the flattened frame before scaling, so every host previews a pending move/paste without mutating the layer. Item #10 onion skinning: `ComposeOnionSkin(frames, index, settings, buffer)` composites N previous/next ghost frames (farthest first, per-side base alpha times `AlphaFalloff^(distance-1)`, clamped at the animation ends — no wrap) under the current frame into a host-agnostic preview buffer, with an onion `RenderToScreenBuffer` overload that scale-blits the same composite; [`OnionSkin.cs`](OnionSkin.cs) holds the validated `OnionSkinSettings`. Item #31 adds the public view API — `SetZoom(zoom)` (clamped, non-finite rejected), anchored `SetZoom(zoom, screenX, screenY)` (bit-identical math to the anchored `OnScroll`), `SetPan` (absolute), and `ResetView` — closing the batch-2 gap where hosts could only zoom through scroll events.
- [`ViewportOverlays.cs`](ViewportOverlays.cs) (item #31) defines the optional designer overlay passes: `ViewportOverlayKind` flags plus one config struct per pass, bundled in the host-owned, reusable `ViewportOverlaySettings`. `RenderToScreenBuffer` overloads taking the settings composite the enabled passes into the screen buffer after the blit — zero allocations in steady state — in a fixed order: checkerboard transparency backdrop (screen-anchored cells composited UNDER the content, inside the canvas area only; moved here from the Avalonia shell so every host renders the same backdrop, and inside-canvas pixels come out opaque), pixel grid (1-pixel lines on the blit's exact source-boundary columns/rows, only at zoom ≥ `MinZoom`), tile grid (`TileSize`-cell boundaries when a tileset/map context is active — the host stamps `TileGrid.TileSize` per render, 0 = inactive), hitbox rectangles (`Frame.Hitboxes` scaled by `PixelsPerUnit`, outlined in the deterministic per-key `GetHitboxKeyColor`; degenerate/non-finite boxes skipped), attachment outlines (bounds of each attachment resolvable from `AttachmentSources`, placed with the export flatten's pivot/flip math), and pivot markers (crosshairs on the optional explicit host pivot and on every attachment anchor — the placed sub-element pivots; frames carry no pivot of their own). Enabled passes validate their configs loudly; grid/backdrop passes replicate the `ScaleBlitNearestNeighbor` mapping so they never leak past the canvas area, while object passes clip to the screen only (attachments may hang off-canvas). Defaults live in `EFYVLabyrinthConfig.LabyMake.Overlay`.
